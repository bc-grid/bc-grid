# v0.6 Migration Notes

This document collects breaking-or-near-breaking consumer-facing changes that land in the v0.6 release line. Each section documents the change, why it's happening, and what consumers need to do (if anything).

---

## Layout architecture pass â€” sticky positioning + single scroll container

PR (a) of [`docs/design/layout-architecture-pass-rfc.md`](../design/layout-architecture-pass-rfc.md) collapses the previously-separate header viewport (`.bc-grid-header-viewport`) and body scroller (`.bc-grid-scroller`) into a single `.bc-grid-viewport`. Header rows become `position: sticky; top: 0` siblings of the body canvas; pinned cells use `position: sticky; left: 0 / right: 0` lane wrappers. The JS-driven scroll-sync helpers (`syncHeaderRowsScroll`, `headerScrollTransform`, `pinnedTransformValue`, `headerViewportStyle`, `autoHeightHeaderViewportStyle`) are deleted.

### What this means for consumers

#### 1. CSS class hard-rename: `.bc-grid-scroller` â†’ `.bc-grid-viewport`

The body scroll container is renamed. Per RFC Â§10 Q2 (ratified), there is **no alias** â€” consumer style overrides keying off `.bc-grid-scroller` need to update to `.bc-grid-viewport`.

Replace any consumer overrides such as:

```css
/* before (v0.5 and earlier) */
.bc-grid-scroller {
  scrollbar-gutter: stable;
}

/* after (v0.6) */
.bc-grid-viewport {
  scrollbar-gutter: stable;
}
```

The structural element is the same `<div ref={viewportRef}>` â€” only the class name changes. CSS overrides that match other class names (`.bc-grid-canvas`, `.bc-grid-header`, `.bc-grid-cell`, `.bc-grid-cell-pinned-left`, `.bc-grid-cell-pinned-right`) keep working unchanged.

A new `.bc-grid-header-band` wrapper class also appears around the header rows. Consumers who want to style the sticky header band as a whole (e.g. add a bottom border or background tint that spans the full header chrome) can hook into it directly.

#### 2. iOS Safari sticky positioning + transformed ancestors

`position: sticky` does not compose with a transformed ancestor in the DOM tree on Safari (and other browsers historically). The grid's own root has `position: relative` (no transform), so the grid itself is fine in isolation. **Consumer apps that wrap `<BcGrid>` or `<BcServerGrid>` in a transformed ancestor â€” CSS `transform`-based slide-in animations, drawer transitions, animated route panels â€” may see broken sticky composition** (the header strip and pinned columns "scroll away" with the body instead of pinning).

Workarounds:

- Avoid CSS `transform` on any ancestor of the grid for the duration of grid render. Animate via `opacity`, `clip-path`, or `translate` (the standalone CSS `translate` property does not break sticky descendants in Chromium / Firefox but historically has on Safari â€” verify on your target browsers).
- If a transform must persist, mount the grid in a portal at the document root so the transformed ancestor is bypassed.
- For drawer transitions specifically, prefer the drawer's `transform: translate3d` to be on the drawer ITSELF, not on a parent that also contains the grid.

This is a browser-engine constraint, not a `bc-grid` policy. Once the consumer's ancestor tree is transform-free, sticky composition works automatically.

#### 3. Internal helpers removed (off-contract)

These exports from `@bc-grid/react`'s internal `gridInternals` module are deleted in PR (a):

- `headerScrollTransform`
- `headerViewportStyle`
- `autoHeightHeaderViewportStyle`
- `syncHeaderRowsScroll`
- `pinnedTransformValue`
- `scrollerStyle` (renamed to `viewportStyle`)

None of these were in the api-surface manifest (`tools/api-surface/src/manifest.ts`) â€” they were framework internals for cross-module use within `@bc-grid/react`. Consumers reaching for them are off-contract and absorb the deletion. If you depended on `pinnedTransformValue` for a custom range-selection overlay or similar, the function's algebra still applies â€” copy the 4-line implementation locally:

```ts
function pinnedTransformValue(
  pinned: "left" | "right" | null,
  scrollLeft: number,
  totalWidth: number,
  viewportWidth: number,
): string | undefined {
  if (pinned === "left") return `translate3d(${scrollLeft}px, 0, 0)`
  if (pinned === "right") {
    return `translate3d(${scrollLeft + viewportWidth - totalWidth}px, 0, 0)`
  }
  return undefined
}
```

(The `@bc-grid/react` internals own the lane wrappers' sticky CSS; consumers do not need this transform anymore unless they're rendering their own content adjacent to the grid that has to track the body's horizontal scroll.)

## Visual contract consolidation â€” `data-bc-grid-edit-state`

Per [`docs/coordination/v05-audit-followups/worker3-editors-and-validation.md` Â§4](../coordination/v05-audit-followups/worker3-editors-and-validation.md): the v0.5 cell-state visuals lived in three CSS rule blocks (cell-level via `data-bc-grid-cell-state` + `aria-invalid`; editor input via `data-bc-grid-editor-state` + `aria-invalid`; editor portal via `data-bc-grid-editor-state`) with overlapping but slightly different contracts. Consumer themes had to override the same colour in three places to keep them in sync; the `aria-invalid` styling and the bc-grid-private `data-*` attribute styling could drift apart depending on which selector a consumer override targeted.

v0.6 collapses this onto a single canonical attribute + token cascade. The cell, editor input, and editor portal all stamp `data-bc-grid-edit-state="..."` (one of `idle` / `dirty` / `pending` / `error`); CSS targets that attribute via one rule per state. The legacy attributes (`data-bc-grid-cell-state`, `data-bc-grid-editor-state`) AND `aria-invalid` styling are preserved as **deprecated aliases for one release** â€” they continue to apply through v0.6 so consumer overrides keying off them keep working. **Removal scheduled for v0.7.**

### What this means for consumers

#### 1. New canonical attribute + tokens

`@bc-grid/react` now stamps `data-bc-grid-edit-state` on the cell wrapper, the editor portal wrapper, and every built-in editor input. Values: `idle | dirty | pending | error`. The legacy attributes are still stamped alongside (dual-attribute), so the migration is non-breaking.

`@bc-grid/theming` defines four new CSS custom properties that drive every edit-state surface:

```css
:root {
  --bc-grid-edit-state-error-stroke: var(--bc-grid-invalid);
  --bc-grid-edit-state-error-fg: var(--bc-grid-invalid);
  --bc-grid-edit-state-error-bg: color-mix(in srgb, var(--bc-grid-invalid) 14%, var(--bc-grid-bg));
  --bc-grid-edit-state-error-flash-bg: color-mix(in srgb, var(--bc-grid-invalid) 25%, transparent);
  --bc-grid-edit-state-pending-stroke: var(--bc-grid-dirty);
  --bc-grid-edit-state-dirty-stroke: var(--bc-grid-dirty);
}
```

Consumer themes override one variable and the cell stripe, the editor input border, the validation popover, AND the validation flash all follow:

```css
/* Tint everything BusinessCraft-red without touching three selectors */
.your-theme {
  --bc-grid-edit-state-error-stroke: hsl(355 80% 40%);
  --bc-grid-edit-state-error-bg: hsl(355 80% 96%);
}
```

#### 2. Recommended migration path

If your overrides hit the legacy attributes or `aria-invalid`:

```css
/* before (v0.5 â€” still works in v0.6, removed in v0.7) */
.bc-grid-cell[aria-invalid="true"] {
  background: pink;
}
.bc-grid-cell[data-bc-grid-cell-state="error"] {
  background: pink;
}
.bc-grid-editor-input[data-bc-grid-editor-state="error"] {
  background: pink;
}

/* after (v0.6+, single canonical selector for every surface) */
[data-bc-grid-edit-state="error"] {
  background: pink;
}
```

The `aria-invalid` attribute itself is **still stamped** on cells + inputs â€” it remains the AT contract (assistive tech reads it). What's deprecated is *styling on the `aria-invalid` selector*; new styling should target `data-bc-grid-edit-state` instead.

#### 3. What stays the same

- The `data-bc-grid-error-flash="true"` attribute (worker3 #407 audit P1-W3-4) is unchanged. The flash keyframe now reads its colour from `--bc-grid-edit-state-error-flash-bg` so consumer tinting composes correctly.
- The `editorControlState` helper is still exported from `@bc-grid/editors` â€” it's the resolver every editor uses. Custom editor implementations can keep using it directly OR switch to the new `editorStateAttrs({ error, pending })` helper that returns the dual-attribute pair as a spreadable object.
- All 8 built-in editor `kind` discriminators (`text`, `number`, `date`, `datetime`, `time`, `checkbox`, `select`, `multi-select`, `autocomplete`) are unchanged.
