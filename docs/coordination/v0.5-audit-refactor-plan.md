## Release Thesis

`v0.5.0` exists because the audit found that bc-grid's engineering foundations are exceptional but its **public API is too low-level** for the BusinessCraft ERP migration. Every consumer grid in `~/work/bsncraft` wires ~30 controlled-state props and a hand-rolled server pagination state machine; bsncraft already carries 2,142 LOC of wrapper code across 5 wrappers; zero of the four hero use cases are built yet.

`v0.5.0` ships the turnkey hooks, imperative API expansion, and four hero-use-case spike grids that prove a typical CRUD grid in the BusinessCraft ERP becomes ~30 lines of consumer code instead of ~200. It does not add new features beyond what the audit identified as P0/P1 ergonomics. It does not refactor packages or break the public API surface for advanced consumers — the existing controlled-prop API stays.

This is **alpha-quality work shipped before the migration scales**. Every grid built on the old wrapper layer is a place a future contributor can re-bug a state machine that should be turnkey. Fixing now is dramatically cheaper than fixing across N copies.

## Why now (not v1.0 or post-1.0)

- bsncraft demo target is 2026-05-30. The demo will go out on whatever wrapper layer exists. Range/clipboard helpers shipped without ergonomics fixes still require 9-`useState` server orchestration per grid; the demo *feels* like a windows-client downgrade.
- Ergonomics changes are mostly **additive** (new hooks, new apiRef methods) and don't break existing consumers. They can ship on the same trunk that v0.4 just landed.
- The four hero use case spikes are the early-warning system for v1.0 scope. Each spike will surface 3–10 missing patterns. Better to find them in spike form (one apps/examples grid each) than mid-bsncraft-migration on real domain code.

## Hero Use Case Spike Specs

Each spike is **one file in `apps/examples/src/`**. Each must:

- Render a believable ERP grid with deterministic seed data (no live backend).
- Stay under 100 LOC of consumer code (imports + JSX + types). Anything that pushes over is a finding to surface in the spike's PR description.
- Use the new `useBcGridState` and (where server-backed) `useServerPagedGrid`.
- Document any pattern the spike had to wrap or work around in a `## Findings` section at the bottom of the example file as a JSDoc block. Those findings feed into v0.6+ planning.

| Spike | Owner | Patterns expected to surface |
|---|---|---|
| Sales estimating | worker3 | money column with currency formatting, dependent-cell recompute on commit, Excel paste with type coercion |
| Production estimating | coordinator | outline rendering for parent/child rows, drag-to-reorder with hierarchy constraints, multi-row "apply edit to N selected" |
| Colour selection | worker3 | swatch-aware lookup option rendering, recently-used section, create-new-inline |
| Document management | coordinator | file column type, drag-drop upload onto a row, thumbnail preview, bulk action toolbar |

The spikes are not production grids. They are **early-warning instruments** — discovery vehicles for what `v0.6.0` (range/clipboard) and beyond will need to support.

## Scope Boundaries

**In scope:**
- New hooks (`useBcGridState`, `useServerPagedGrid`, possibly companions)
- `BcGridApi` expansion (focus/scroll/edit/filter/getters)
- Four hero spike grids in `apps/examples/`
- Cheap P1s (test-import lint rule, Cmd+F prop, fit prop, optional filter prop)
- One bsncraft grid migration as validation
- Audit synthesis doc

**Explicitly out of scope (defer to v0.6+):**
- Peer-deps refactor for `editors`/`enterprise` (P1-C1 in coordinator audit) — risky for tight sprint, plus breaks consumer install ergonomics if rushed
- Callback-shape standardization (P1-C4) — pure API churn, breaks consumers without delivering ERP value yet
- Naming inconsistency cleanup (P2-C1) — same reasoning
- Range/clipboard work (now `v0.6.0`)
- Charts (post-1.0 always)

**Out of scope for any release pre-1.0:**
- Charts integration
- RTL
- Spreadsheet-class formula editing
- Bug-for-bug AG Grid edge-case parity
- Mobile-first (touch fallback only)
