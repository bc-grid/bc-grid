# v0.1-alpha Release Plan — making `main` shippable as a coherent alpha

**Owner:** c2 (auditor + coordinator)
**Date:** 2026-04-30
**Status:** Plan; supersedes nothing. Runs **in parallel** with `docs/coordination/v1-parity-sprint.md`.
**Authoring context:** `audit-c2-003.md` found the v0.1 public surface is wider than its implementation (4H + 14M + 5L findings). This plan resolves them so `main` can ship as a coherent v0.1-alpha release while the v1 parity sprint continues on its own track toward v1.0.

---

## What "v0.1-alpha unbroken" means

A consumer who installs `@bc-grid/react@0.1.0-alpha.1` should be able to:

1. Render `<BcGrid>` and have every documented prop either **work as documented** or **be absent from the type system** (no silent typed-but-unwired surface).
2. Read `docs/api.md`, see the v0.1-alpha column "supported / reserved", and know exactly what's safe to call.
3. Hit no silent footguns — every public method either does something useful, or fails loudly with a clear error.
4. Hit no surprises: the 6 placeholder packages don't appear in the published v0.1-alpha surface.
5. Use the read-only feature set (filter / sort / pinned cols / row selection / live regions / theming) confidently.

What v0.1-alpha **is not**:
- Not v1.0 — that's the parity sprint goal.
- Not editing — Track 1.
- Not server-row-model beyond `paged` mode — Track 3.
- Not range selection / clipboard — Track 2.
- Not aggregations / pivots / chrome / charts — Tracks 4 / 5 / 7.
- Not all 8 filter types — only `text` (with `contains` operator) until Track 6.

The boundary: **everything currently merged on `main` plus the small fixes below**, with the typed surface narrowed to match.

## Acceptance criteria

A v0.1-alpha tag is shippable when ALL of these are true:

- [ ] `bun install && bun run type-check && bun run lint && bun run test && bun run build && bun run test:e2e && bun run api-surface` all green from a clean checkout.
- [ ] **No `BcGridProps` field is typed-but-unread.** Every declared prop has a code path that consumes it (or is removed from the v0.1-alpha types).
- [ ] **No `BcGridApi` method is an empty no-op.** Every method either does work or throws a clear error.
- [ ] **No filter type silently filters out everything.** Unrecognized filter shapes warn + fall through to "match" (don't silently drop rows).
- [ ] **The 6 placeholder packages are not in the v0.1-alpha publish set.** Source stays in the monorepo for the v1 parity sprint to fill in; npm publish skips them.
- [ ] **`api.md` status block reads "v0.1-alpha"** and every reserved-for-Q2/Q4/etc. property is annotated with `@unimplemented` JSDoc.
- [ ] **`README.md`** describes what v0.1-alpha includes + a "what's coming in v1.0" forward-pointer to the v1 parity sprint.
- [ ] **Smoke perf bars** from `design.md §3.2` enforced on every PR (the `smoke-perf-ci` task).
- [ ] **Bundle size budget** enforced on every PR (the `bundle-size-ci-gate` task — already in flight as #59).
- [ ] **No regression** on the AR Customers demo's e2e suite.
- [ ] **An `audit-c2-004.md` clean of severity-H + severity-M findings** (the v0.1-alpha gate audit).
- [ ] **Release notes** documented in `docs/release-notes/v0.1.0-alpha.1.md` with explicit "what works / what's reserved / what's planned for v1.0".

When all 11 boxes are checked, tag `v0.1.0-alpha.1` and (optionally) publish.

## Phase 1 — Surface narrowing (1 sequential PR; lands first)

**Goal:** strip every typed-but-unwired prop / method from the v0.1-alpha public surface so the surface matches the implementation.

**Why sequential:** this is one cohesive change to `packages/core/src/index.ts` + `packages/react/src/types.ts` + `tools/api-surface/src/manifest.ts` + `apps/examples/src/App.tsx` (if any unwired props are demonstrated there). Splitting it would create merge conflicts on the same files. **Single-owner PR.**

### What lands in this PR

#### Strip from `BcGridStateProps` (in `@bc-grid/core`)
- `expansion`, `defaultExpansion`, `onExpansionChange` — moved to `BcGridStatePropsV1` (reserved type for v1 parity sprint to extend).
- `groupBy`, `defaultGroupBy`, `onGroupByChange` — same.
- `page`, `defaultPage`, `pageSize`, `defaultPageSize`, `onPaginationChange` — same.

#### Strip from `BcGridProps` (in `@bc-grid/react`)
- `pagination`, `pageSizeOptions` — removed.
- `groupableColumns`, `groupsExpandedByDefault` — removed.

#### Narrow `BcColumnFilter.type` (in `@bc-grid/core`)
- From `"text" | "number" | "date" | "set" | "boolean" | "custom"` to `"text"` only at v0.1-alpha.
- Re-widen in the Track 6 `filter-registry-impl` PR per `filter-registry-rfc` (#48).
- The matching `ServerColumnFilter.type` widens identically — keep that one wide for server-grid consumers (server-side filters exist independently of client UI), but client UI rejects non-text at compile time.

#### Narrow `BcServerGridProps` (in `@bc-grid/react`)
- v0.1-alpha exports `BcServerPagedProps` only.
- `BcServerInfiniteProps` and `BcServerTreeProps` remain in source (Track 3 will fill them in) but **don't appear in the discriminated union** at v0.1-alpha.
- `BcServerGridProps` becomes a type alias for `BcServerPagedProps` until v1.0.

#### Strip from `BcGridApi` (in `@bc-grid/core`)
- `expandAll()` and `collapseAll()` — removed at v0.1-alpha. Re-add when `group-by-client` (Track 0) lands.

#### Strip from `BcServerGridApi`
- `invalidateServerRows`, `retryServerBlock`, `getServerRowModelState` — keep on the type because Track 3 #60 (in flight) will wire them. But add `@unimplemented` JSDoc until #60 + follow-ups merge.

#### Strip from `column.tooltip`
- Keep declared on `BcGridColumn` (Phase 5.5 `tooltip-rendering` is `[ready]` and small). Implement in Phase 3 instead of removing.

#### Strip 6 placeholder packages from publish set
- Edit `package.json` workspaces or per-package `package.json` `private: true` until each has an implementation.
- **Stay in the monorepo source tree** — the v1 parity sprint Tracks 1, 3, 4, 6 will fill them in.
- `tools/api-surface/src/manifest.ts` keeps them in `mode: "planned"` for tracking; no consumer-facing publish.

#### Update `api.md`
- Status block: "v0.1-alpha — frozen API surface; implementation matches type system."
- Every removed prop documented with explicit "moved to Track N" / "reserved for v1.0".
- Every kept-but-unimplemented prop annotated `@unimplemented Q2` / etc.

| **N1** Single PR: surface narrowing across core / react / api.md / manifest / examples | L (4-5 hours focused) | c1 (touches `grid.tsx`-adjacent types) or c2 (coordination domain) |

```
Phase 0 (docs)        ────────────────────────────────────────────►

                          │

Phase 3 (small impls)     ───────►   I1 ‖ I2 ‖ I3 ‖ I4
                          │
                          ▼
Phase 4 (quality)         ───────►   Q1 (in flight) ‖ Q2 ‖ Q3 ‖ Q4
                                                            │
                                                            ▼
Phase 5 (release)                                           ───►   R1 → R2 → R3 → R4 → R5
```

Phase 1 lands first as a single PR. Phases 2/3/4 run in parallel after that — no cross-phase dependencies. Phase 5 lands once Phase 2-4 are all merged.

**Critical-path bottleneck:** Phase 1 is single-owner and touches surface types. Until it lands, no Phase 2 PR can proceed (they assume the narrower surface). **Estimated Phase 1 duration: 4-5 hours.**

After Phase 1: 13 parallel PRs across Phases 2-4. With 4 implementer agents at typical velocity (~1 PR per agent per 2 hours), all 13 land in ~6-8 hours. Phase 5 is then ~1-2 hours.

**Total elapsed time, optimistic, with full agent parallelism: 12-15 hours.**
**Total elapsed time, realistic, with 50-75% agent utilisation: 20-30 hours.**

## Parallelism with the v1 parity sprint

This alpha-release plan and the v1 parity sprint are **not** in conflict — they're complementary:

- **The alpha plan** stabilises today's `main` so it can ship.
- **The v1 parity sprint** builds the rest (editing, range, server-infinite, aggregations, pivots, chrome) on top. Charts are post-1.0.

Specifically:
- Phase 1's surface narrowing **does not block** any v1 parity sprint Track. Tracks 1/3/4/6/7 add NEW exports; the narrowed surface is the floor they extend.
- Phase 3's small impls are already in queue.md as `[ready]` Phase 5.5 tasks; the alpha plan just promotes them to alpha-blockers.
- Phase 4's quality gates apply to ALL future PRs, including v1 parity sprint PRs.
- Phase 0's docs include forward-pointers to the v1 parity sprint so consumers know what's coming.

When v1.0 ships (end of v1 parity sprint), it follows v0.1-alpha-N as a normal semver progression: `0.1.0-alpha.1` → `0.1.0-alpha.2` (more parity sprint progress) → ... → `1.0.0-rc.1` → `1.0.0`.

## Open questions

### Should the 6 placeholder packages be private OR removed entirely from monorepo at v0.1-alpha?
**Decision: stay in monorepo, mark `private: true`.** The v1 parity sprint will fill them in; deleting + re-creating creates churn. `private: true` means they don't publish to npm but stay in the workspace.

### Should v0.1-alpha bundle the AR Customers demo into the published assets?
**Decision: no.** `apps/examples` stays separate. Consumers wanting a demo clone the repo; the published packages are the library only.

### How to communicate "v0.1-alpha is read-only" without sounding underselling?
**Decision: lead with what works.** The README header is "bc-grid v0.1-alpha — high-performance read-only data grid for React" not "alpha placeholder release". Specifics come after.

### Do we npm-publish v0.1-alpha or stay private?
**Decision (open):** depends on whether consumers want to install it. If internal-only, skip publish; tag the git ref. If external, publish with `--tag alpha` so it's not the default install. Defer to JohnC.

### What if Phase 1 reveals a prop that's actually wired but I missed?
**Mitigation:** Phase 1's PR description lists every prop being removed. Reviewer checks. Ten minutes of careful review beats post-merge revert.
