## 1. Where bc-grid is already better or more ERP-focused

### Editing pipeline

1. **Async validation with `AbortSignal` is first-class.** `column.validate(value, row, signal)` returns `BcValidationResult | Promise<BcValidationResult>`; the controller maintains a per-cell `AbortController` and cancels superseded validations. AG Grid's documented validator path is synchronous (`newValueHandler` / `valueSetter` returns boolean or value); async validation is consumer-owned (wrap your own debounced fetch + reject the commit).
2. **Stale-mutation guard via `mutationId`.** Every commit stamps the cell entry with a monotonic `mutationId`; the resolve and reject paths both compare the captured id against the entry's current id and silently bail when superseded by a re-edit. AG Grid's docs do not call out an equivalent guarantee; consumers using the undo stack effectively rely on it but the public docs leave the semantics on Promise-based commits informal.
3. **Overlay + `pruneOverlay()` cleanup on `data` prop update.** `<BcGrid>` carries an internal `BcEditOverlay.patches` map, then walks it after `data` prop changes and clears entries whose canonical row value matches the patch. AG Grid relies on the consumer to drive transactional row updates (`applyTransaction` / `applyTransactionAsync`) ÔÇö bc-grid's overlay model is implicit.
4. **Portal-aware click-outside is composable.** `editorPortal.tsx` listens at `document.pointerdown` (capture) and walks `closest('[data-bc-grid-editor-root], [data-bc-grid-editor-portal]')` so editors composed of arbitrary popovers (date pickers, autocomplete dropdowns, custom shadcn/Radix-style portals) can opt in by attribute. AG Grid's `cellEditorPopup: true` / `cellEditorPopupPosition` is binary and editor-typed; bc-grid's marker pattern composes with any portal library a consumer brings.
5. **Editor framework contract is leaner.** `BcCellEditor` is `{ Component, kind?, prepare? }` plus `BcCellEditorProps` with `commit` / `cancel` callbacks. AG Grid's vanilla `ICellEditor` requires implementing `init` / `getValue` / optional `isCancelAfterEnd` / `isCancelBeforeStart` / `focusIn` / `focusOut` / `afterGuiAttached` / `destroy`; the React adapter adds `useGridCellEditor` boilerplate. bc-grid's surface is roughly half the contract.
6. **Typed-commit + `moveOnSettle` in one call.** `commit(value, { moveOnSettle: "right" })` lets numeric / date / select-style editors land a typed value AND the user's movement directive in one call (PR #181 / `editor-typed-commit`). AG Grid's editors generally call `stopEditing()` separately; the framework reads `getValue()` and applies its own "next cell" rules.
7. **Source attribution on commit events.** `BcCellEditCommitEvent.source: "keyboard" | "pointer" | "api"` ÔÇö useful for ERP audit logging. AG Grid's `CellValueChangedEvent` carries `source: "edit" | "paste" | "api" | ÔÇŽ` but the granularity differs.

### Accessibility

8. **Explicit ARIA contract pinned in tests.** bc-grid emits `aria-current="true"` on the editing cell, suspends `aria-activedescendant` while DOM focus lives in the editor input, attaches `aria-describedby` to a `useId()`-keyed visually-hidden error span (cell side and input side), and `aria-label`s the editor input from `column.header`. The accessibility-rfc enumerates this and `wcag-code-pass` (PR #191) added unit tests pinning the helper outputs and the theming CSS contract for `prefers-reduced-motion` / `forced-colors` / `pointer: coarse`. Public AG Grid docs cover ARIA at a higher level ÔÇö role, basic keyboard ÔÇö but don't pin the editor-specific edge cases bc-grid does (suspended activedescendant during edit, aria-current on the underlying gridcell).
9. **Live-region templates fully localised through `BcGridMessages`.** Sort, filter, selection, edit-committed, edit-validation-error, edit-server-error all flow through consumer-overridable templates; `wcag-code-pass` extended this with filter-input strings. AG Grid has `localeText` but the per-event templates (cell-edit-rejected, server-rollback) are less granular; consumers typically intercept the events and announce themselves.
10. **Polite-region 250ms tail debounce.** Tab-through-N-cells gets one announcement at the tail rather than N at speech-engine pace. Assertive announcements are deliberately undebounced. AG Grid's docs describe announcements informally; debounce strategy is consumer-owned.
11. **Reduced-motion / forced-colors are theming-layer first-class.** `packages/theming/src/styles.css` ships `@media (prefers-reduced-motion: reduce)` zeroing transitions/animations/scroll-behavior, `@media (forced-colors: active)` mapping to system colours with a real `outline` (not `box-shadow`) on the active cell, and `@media (pointer: coarse)` raising hit targets to 44px. CSS-string assertions in `packages/theming/tests/theming.test.ts` keep this from drifting. AG Grid's themes ship reduced-motion-respectful CSS too; the forced-colors / pointer: coarse contract is less explicitly documented.

### Performance / virtualizer

12. **Virtualizer in-flight retention for the editor row + column.** `editorPortal` calls `virtualizer.beginInFlightRow(rowIndex)` + `beginInFlightCol(colIndex)` on mount; reference-counted, idempotent. Scroll / sort / filter during edit can't unmount the editor's DOM. AG Grid retains the editor in DOM by similar mechanics but the explicit reference-count + per-row/per-col handles are bc-grid's purpose-built shape.
13. **`focusRef` timing pinned to `useLayoutEffect`.** Editors must assign `focusRef.current = inputRef.current` in `useLayoutEffect` (not `useEffect`) so the framework's mount-focus call sees the assigned ref ÔÇö child layout effects fire before parent layout effects. The custom-editor recipe doc (`apps/docs/src/pages/editor-custom-recipe.astro`) calls this out explicitly. The equivalent timing trap exists in any React framework but AG Grid's docs gloss over the React-specific edges.

### ERP posture

14. **`<BcEditGrid>` ships an action column out of the box.** `<BcEditGrid onEdit onDelete extraActions linkField detailPath>` renders a pinned-right actions column with destructive-action disabling while the row has a pending in-flight commit (`bc-edit-grid-complete`, #166). AG Grid leaves the action column to the consumer (typical pattern: a custom `cellRenderer` with buttons; consumer wires the disable logic).
15. **`linkField` first-class.** A typical ERP "row-as-link" pattern (click the customer name Ôćĺ detail page) is shipped via `BcEditGridProps.linkField` + `detailPath`. AG Grid leaves this to a per-column `cellRenderer`.
16. **`valueParser` placement is documented and bypassed for typed editors.** Per editing-rfc, `valueParser` runs only when the editor produces a string AND the column declares a parser. Typed editors (date / select / multi-select / autocomplete / number with internal parse) commit a typed value directly. AG Grid's `valueParser` fires unconditionally on `newValue` strings; typed editors that already commit a typed value end up needing to either bypass or no-op the parser at the consumer level.
17. **Editor catalogue maps to ERP filter-type catalogue.** Same eight built-ins (`text` / `number` / `date` / `datetime` / `time` / `select` / `multi-select` / `autocomplete`) line up with the eight column filter types (`text` / `number` / `number-range` / `date` / `date-range` / `set` / `boolean` / `custom`) ÔÇö consumers pick filter + editor in tandem rather than juggling two separate vocabularies.

## 3. Gaps (P0 / P1 / P2 / P3)

Severity legend: **P0** blocks an ERP cutover; **P1** strong wishlist for v1.0; **P2** nice-to-have polish; **P3** lower priority / out of v1 scope per editing-rfc.

### P0

- **Undo / redo for cell edits.** AG Grid documents `undoRedoCellEditing` + `undoRedoCellEditingLimit` plus `Ctrl+Z` / `Ctrl+Y` keyboard handling. bc-grid has none. The editing-rfc explicitly defers this to post-1.0 ("Undo/redo. Post-1.0; needs a separate state machine RFC. The commit pipeline below is undo-friendly (immutable patches) but no UI ships in this sprint.") For an ERP that uses bc-grid as a daily entry surface, keyboard undo is table-stakes ÔÇö users expect it.
- **Bulk paste from Excel into a range.** AG Grid documents the clipboard contract: `Ctrl+V` parses TSV from the clipboard, applies `valueParser` per column, and atomic-applies / rolls back. bc-grid has the read side queued (Track 2 `clipboard-paste-from-excel`, `[blocked: depends on clipboard-copy-tsv-html]`) but nothing has landed. Without it the AR-Customers-style "paste a column of opening balances" flow falls back to per-cell editing ÔÇö a real ERP rollout blocker.
- **Range copy (TSV + HTML).** Demo-critical per the queue (`range-state-machine` + `visual-selection-layer` + `clipboard-copy-tsv-html`). Some pieces are landing per the Track 2 status; until the trio merges, range copy isn't usable. AG Grid copies with `Ctrl+C` out of the box.

### P1

- **Full-row editing mode.** AG Grid's `editType: "fullRow"` + `cellEditor` per column; whole row enters edit, Tab traverses editable cells in the row, Enter commits the row's accumulated changes atomically, Esc cancels the whole row. bc-grid is single-cell-at-a-time (per editing-rfc ┬žConcurrency). For ERP "approval workflow" rows where 4-5 fields move together (status + date + reason + amount), full-row editing is the natural shape.
- **Multi-line text editor (`agLargeTextCellEditor` analogue).** ERP "Notes" / "Comments" columns are routine; bc-grid ships only single-line `editor-text`. The editing-rfc ┬žOpen questions notes "Multi-line text editor (Excel Alt+Enter)? Out of v1 scope. Single-line text input only. Multi-line cell rendering is supported (rowHeight + word-wrap CSS) but the editor is single-line. Can add `editor-textarea` post-1.0." Worth pulling forward.
- **Editor popup mode (break out of cell bounds).** AG Grid's `cellEditorPopup: true` lets wide editors render larger than the cell (e.g., rich-text, calendar with side-by-side months). bc-grid's editor portal is positioned absolutely over the cell rect; nothing today positions it above/below or out of the cell's overflow box. The `data-bc-grid-editor-portal` marker hints at a path for popovered editors but there's no formal "render bigger than the cell" API.
- **Fill handle.** AG Grid Enterprise; ERP users routinely "drag this opening-balance value down 50 rows". Already in queue (Track 2, `[blocked: depends on clipboard-paste-from-excel]`). Pulled forward to v0.4 if the consumer rollout demands it.
- **Per-column `flashOnEdit` granularity.** Today bc-grid has grid-wide `BcGridProps.flashOnEdit?: boolean`. AG Grid offers `enableCellChangeFlash` at the column level so a numeric "balance" column can flash without polluting the "tradingName" column. Small additive change.

### P2

- **`onCellEditStart` event with veto.** AG Grid emits `cellEditingStarted` / `cellEditingStopped`. Consumer can use the start event to gate (e.g., "this user can't edit credit-hold rows"). bc-grid's column.editable predicate handles per-row gating but there's no event-style hook for cross-row conditions or for analytics. Additive to `BcEditGridProps`.
- **`row-revert-api` / `BcGridApi.revertRow(rowId)`.** Pairs with full-row editing; lets a consumer drop all overlay patches + edit entries for a row in one call. Today the pattern is to mutate the consumer's `data` prop and let `pruneOverlay` clean up ÔÇö works but is two-step. AG Grid's `applyTransaction({ update: [{ rowId, original }] })` is roughly equivalent; bc-grid could ship the imperative shortcut.
- **`aria-readonly` on read-only cells in editable grids.** Already noted in `wcag-code-pass`'s impl report (`docs/design/a11y-impl-report.md` "Per-cell `aria-readonly` in editable grids"). NVDA in particular treats `aria-readonly` differently from `aria-disabled`. Public AG Grid docs assert this attribute on the read-only-in-mixed-grid case.
- **Keystroke-time validation hooks.** AG Grid editors can validate per keystroke (custom cell editor with internal state). bc-grid's `column.validate` only fires at commit. Trade-off: the commit-only model is simpler and matches the editing-rfc's intent. If we add keystroke validation it should be opt-in via an editor-side hook so simple cases stay simple.
- **Editor-popup positioning helpers.** Once popup mode lands (P1), helpers for "above" / "below" / "auto" placement; flip-on-overflow logic; offset from cell rect. Internal primitives.

### P3

- **Formula editing (Excel `=A1+B1`).** Out of v1 scope per editing-rfc ┬žNon-Goals. AG Grid Enterprise has limited formula support; not a v0.4 candidate.
- **Inline cell-validation tooltip on rejection.** Today bc-grid surfaces validation errors via `aria-invalid` + assertive live region + `aria-describedby` to a hidden span. Adding a visible tooltip popover on the cell would be polish. AG Grid's docs show a pattern for this but it's consumer-owned.
- **Image / file cell editor.** Niche for ERP; not in scope.
- **Drag-and-drop cell ordering inside an edit row.** Niche.
- **`stopPropagation` semantics for keyboard inside custom editors.** bc-grid's custom-editor recipe documents `event.stopPropagation()` + typed `commit({ moveOnSettle })`; AG Grid's vanilla cellEditor has more involved key-passing rules. bc-grid's surface is already simpler ÔÇö call this resolved.

## 4. Concrete bc-grid-native follow-up tasks

Suggested queue.md entries. Names follow the existing kebab-case convention; effort sizes use the queue's S / M / L bands.

### Track 1 (editing) follow-ups

- **`undo-redo-cell-editing`** (P0, **L**) ÔÇö undo/redo state machine for cell edits, keyed by `mutationId`. `Ctrl+Z` / `Ctrl+Y` keyboard handling at the grid root. Configurable depth (`BcGridProps.undoRedoLimit?: number`, default 50). Pairs with the existing overlay map (every commit is already an immutable patch). Needs an RFC because the commit pipeline interacts with the server-row-model `pendingMutations` and the stale-mutation guard. Estimate: 1 RFC + 1 framework PR + 1 docs PR.
- **`full-row-editing-mode`** (P1, **M**) ÔÇö opt-in `BcEditGridProps.editType?: "single-cell" | "full-row"`; in full-row mode, Tab traverses editable cells in the row, Enter commits the row's accumulated patches atomically, Esc cancels the whole row. Needs careful interaction with the existing `editingCell` state and the `aria-current` semantics (row-level rather than cell-level on entry). Estimate: 1 RFC addendum + 1 framework PR + 1 example.
- **`editor-large-text`** (P1, **M**) ÔÇö `<textarea>`-based multi-line text editor with auto-popup-mode + `Esc` / `Ctrl+Enter` commit semantics. Depends on `editor-popup-mode` (next item) for the popup positioning.
- **`editor-popup-mode`** (P1, **M**) ÔÇö `BcCellEditor.popup?: boolean | { placement?: "above" | "below" | "auto"; offset?: number }`. Editor portal positions the editor as a popover anchored to the cell; default placement is `auto` with overflow flip. Existing `data-bc-grid-editor-portal` marker handles the click-outside + a11y plumbing already.
- **`onCellEditStart-veto`** (P2, **S**) ÔÇö `BcEditGridProps.onCellEditStart?: (event) => boolean | void` event fires before `editor.prepare()`; returning `false` cancels activation. Also expose `cellEditingStarted` / `cellEditingStopped` on the consumer event surface for analytics.
- **`row-revert-api`** (P2, **S**) ÔÇö `BcGridApi.revertRow(rowId): void` / `BcGridApi.revertCell(rowId, columnId): void`. Imperative shortcuts that drop overlay patches + cell edit entries; pair with full-row editing.
- **`per-column-flash-on-edit`** (P2, **S**) ÔÇö `BcReactGridColumn.flashOnEdit?: boolean` overrides the grid-level setting; default falls through to `BcGridProps.flashOnEdit`.

### Track 2 (range / clipboard) ÔÇö already in queue, pulled forward

- **`clipboard-copy-tsv-html`** (P0) ÔÇö already in queue; demo-critical.
- **`clipboard-paste-from-excel`** (P0) ÔÇö already in queue (`[blocked: depends on clipboard-copy-tsv-html]`); pull forward to v0.4 with the rest of the Track 2 stack.
- **`fill-handle`** (P1) ÔÇö already in queue (`[blocked: depends on clipboard-paste-from-excel]`).

### Track 7 / a11y polish

- **`aria-readonly-mixed-grid`** (P2, **XS**) ÔÇö emit `aria-readonly="true"` on body cells that are read-only when at least one column in the grid is editable. Already noted in `docs/design/a11y-impl-report.md` "Per-cell `aria-readonly` in editable grids" as part of the `wcag-deep-pass` follow-up. Cross-link the queue entry.

### Polish / docs

- **`cell-edit-tooltip-on-error`** (P3, **S**) ÔÇö opt-in popover-tooltip showing the validation error on the cell. Pairs with the existing assertive live region; visual reinforcement for sighted users.
- **`editor-popup-large-text-recipe`** (P3, **S**) ÔÇö docs page in `apps/docs` with a worked custom editor for a long-text + popup combination.

## 5. Decisions for v0.4 planning

These are recommendations ÔÇö the architect / coordinator decides the actual cut.

1. **P0 work goes into v0.4.** Specifically `undo-redo-cell-editing` and the Track 2 trio (`clipboard-copy-tsv-html`, `clipboard-paste-from-excel`, plus enough of `range-state-machine` / `visual-selection-layer` to make paste land). Without these the ERP cutover hits friction we know is there.
2. **`full-row-editing-mode` + `editor-large-text` + `editor-popup-mode` are the v0.4 P1 cluster.** They share infrastructure (popup positioning) and together close the bulk of the AG Grid feature delta that an ERP user notices.
3. **P2 items (`onCellEditStart-veto`, `row-revert-api`, `per-column-flash-on-edit`, `aria-readonly-mixed-grid`) are good "fold into the next adjacent PR" candidates** ÔÇö none of them needs an RFC; each is a small additive prop or attribute and can ride alongside the P0/P1 work above.
4. **P3 items stay parking-lot** until v0.5 or later, except `cell-edit-tooltip-on-error` which is a strong consumer-facing polish item once the assertive-region announcements are validated by `wcag-deep-pass`.
